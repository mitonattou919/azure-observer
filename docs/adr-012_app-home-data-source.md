# ADR-012: App Homeダッシュボードのデータ取得方式

- Status: Accepted
- Date: 2026-07-19

## Context

[[adr-010_slack-entry-points-and-channel-routing]]により、App Homeはチャットではなく
「当月コスト」「アクティブアラート一覧」等を表示する常設ダッシュボードとして再定義された。

`CLAUDE.md`の設計思想は「個別リソースAPIを都度実装せず、MCP経由のツール呼び出しに寄せる」ことだが、
これは必ずしも「Foundry Agent（LLM）を経由する」ことまでは要求していない。App Homeの表示データは
自然文の相談ではなく決まった構造化データの取得であるため、取得経路を検討した。

### 選択肢

- **A. Backend（ACAアプリ）がAzure MCPの読み取りツールを直接呼び出す**（Foundry Agentを経由しない）
- B. Agent Aに「今月のコストを教えて」等の問いを投げて回答を取得し、Homeに埋め込む

## Decision

**選択肢A** を採用する。BackendはAzure MCP Serverへの接続経路を持ち、App Home表示用データ
（当月コスト、アクティブアラート一覧）はAgent A（LLM）を経由せず直接取得する。Agent Aは
自然文での相談に特化させる。

取得したデータはライブ取得せず、[[adr-004_in-process-scheduler]]で導入するインプロセス
スケジューラを流用して定期的に取得し、Table Storageにキャッシュする。App Home表示時
（`app_home_opened`イベント）はキャッシュを読むのみとし、その場でAzure MCPを呼び出さない。

キャッシュ更新間隔の目安（暫定値、運用調整の余地あり）:

- アクティブアラート: 15〜30分間隔
- 当月コスト: 1日1回程度（Azure Cost Management自体、当日分の反映に遅延があり、
  高頻度に取得する意味が薄いため）

### 却下した選択肢

- **選択肢B（Agent A経由）**: `CLAUDE.md`の「MCP経由への統合」という記述には忠実だが、
  Home表示のたびにLLM呼び出しが発生しレイテンシ・コストが乗る。また同じ問いでも出力表現が
  微妙に変わりうる（非決定性）ため、構造化データの表示には不向きと判断した
- **毎回ライブ取得（キャッシュなし）**: 実装はシンプルだが、複数ユーザーがHomeタブを
  頻繁に開くとAzure API呼び出しが増え、レート制限や表示遅延につながりうる

## Consequences

- Home表示のたびにLLM呼び出しが発生せず、レイテンシ・コストを抑えられる
- 表示データに最大キャッシュ間隔分の鮮度ズレが生じる（アラートは最大30分程度古い可能性がある）
- Backendが Azure MCP Server への接続経路をAgent経由以外にも持つことになる。`CLAUDE.md` 1章の
  「クライアント側からMCPサーバーを呼ぶ経路は別途Entra ID認証で保護する」という方針は、
  Foundry Agentからの経路に加えてBackend自身からの経路にも同様に適用する必要があり、
  該当箇所の記述を更新する
- キャッシュ更新ジョブが[[adr-004_in-process-scheduler]]のスケジューラに追加されるため、
  同ADRが受容したリスク（マルチレプリカ化時の重複実行）がこのジョブにも同様に適用される
