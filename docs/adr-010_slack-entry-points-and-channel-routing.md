# ADR-010: Slack UI構成とチャネルルーティング

- Status: Accepted
- Date: 2026-07-19

## Context

`CLAUDE.md` 4章・`docs/requirements.md`「Slack UI」節は、App Home上に「チャット履歴表示エリア、
メッセージ入力、送信ボタン」を常設する構成を前提としていた。しかしSlack App Homeの `Home` タブは
`views.publish` でHomeビュー全体を都度再描画する方式であり、通常のメッセージ入力欄のような
継続的な会話UIには不向きである（送信のたびにHome全体を作り直す必要があり、UXが悪化しやすい）。

Issue #2の検討にあたり、この制約を踏まえてSlack UI全体の入口・チャネル構成を再検討した。

### 検討した論点

1. Agent A（相談）への質問導線: App Home常設入力 / DM / チャンネル@メンション、をどう組み合わせるか
2. App Homeの役割: チャットUIのままにするか、別の用途に転用するか
3. 承認（Agent B）・定期通知（Agent C）のチャネル構成: 既存方針（[[adr-006_approval-flow]]）通り
   固定1チャンネルとするが、その1チャンネルを新たに追加するチャット導線と統合するか分離するか

## Decision

### App Homeはダッシュボードとして再定義する

チャットUIとしては使わず、以下を表示する常設ダッシュボードとする。

- 当月コスト
- アクティブアラート一覧
- VM起動申請ボタン（既存Block Kit申請フローへの入口）
- JIT権限付与のプレースホルダー（将来機能。実装が無い段階でも導線の存在だけ示す）

表示データの取得方式は [[adr-012_app-home-data-source]] を参照。

### Agent Aへの質問導線はDM、および「sre」チャンネルでの@メンションに限定する

- **DM**: ユーザーとBotの1:1チャットで質問する。主導線。
- **「sre」チャンネルでの@メンション**: 同チャンネル内でBotにメンションすることでも質問できる。
  対応するのは「sre」チャンネルのみとし、他チャンネルでの@メンションはサポート対象外とする
  （Botを個別に招待すれば技術的には動作しうるが、正式な導線としては提供しない）。
  Azure関連の質問を行うのは組織上SRE室メンバーに限られるという運用実態を踏まえた判断であり、
  汎用的な「Botを招待した任意のチャンネルで質問可能」という構成は採用しない
- App Homeでのチャット入力は提供しない（前述の技術的制約と、ダッシュボード用途への転用のため）

DMとsreチャンネル@メンションの会話は同一の会話文脈を共有しない。詳細は [[adr-011_agent-thread-separation]] を参照。

### 承認・定期通知は「sre」プライベートチャンネル1本に集約する

- [[adr-006_approval-flow]]で決定した「固定1チャンネル」を、Agent Bの承認依頼とAgent Cの定期通知の
  両方の投稿先として使う。新たに通知専用チャンネルを追加しない
- チャンネルは**プライベートチャンネル**とする。ADR-006の「チャンネル参加＝承認権限」という権限モデルが
  成立するためには、参加そのものが制御されている必要がある。パブリックチャンネルでは誰でも自己判断で
  参加でき、参加した瞬間に本人の意図に関わらず承認権限を得てしまうため、モデルの前提が崩れる
- チャンネルへの招待・メンバー管理は運用（誰をSRE室メンバーとして招待するか）で担保し、
  本設計のスコープには含めない

### 却下した選択肢

- **DMを使わず「特定のチャンネル」のみを質問導線にする案**: 当初検討したが、DMは「質問（読み取り専用）」
  に限定する前提であれば副作用のリスクは無く（書き込み系操作はAgent B側で別途承認制により防御されている）、
  1:1でしか聞きにくい質問もあるため、DMも導線として残すことにした
- **質問対応・承認・定期通知を全部同じ1チャンネルにする案**: 承認ボタンの投稿と雑談的な質問が混在し、
  承認者にとってノイズになるため不採用
- **用途ごとに3チャンネルへ分割する案（質問／承認／通知）**: 権限分離は最も厳密になるが、
  チャンネル管理が煩雑になる。「通知を見たい人＝承認権限を持ってよい人（SRE室メンバー）」が
  組織上ほぼ一致するため、承認と通知は分けずに1本化した

## Consequences

- App Home・要件定義書に残る「チャット履歴表示・メッセージ入力」という記述は本ADRにより陳腐化するため、
  `docs/requirements.md` と `.claude/CLAUDE.md` の該当箇所を更新する
- 承認専用チャンネルに質問混じりの投稿・定期通知が同居することになり、承認履歴としての監査ログの
  純度はやや下がる。組織特性（質問者＝SRE室メンバー＝承認候補者）を踏まえて意図的に許容する
- 必要なOAuth Scopesが追加される: `im:history`（DMメッセージ受信）、`app_mentions:read`
  （チャンネル内@メンション受信）、`chat:write`（DM・チャンネルへの送信）、`commands`（`/reset`相当）、
  `users:read`（表示名解決）。App Home（`views.publish`）関連の正確なスコープ名は実装時に
  Slack公式ドキュメントで最終確認する
- 「sre」チャンネルへの招待運用（誰がいつ招待・削除するか）を明文化した運用手順が別途必要になる
  （本ADRのスコープ外、未決事項として残す）
