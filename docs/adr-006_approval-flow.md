# ADR-006: Agent B承認フローの実装方式と承認権限モデル

- Status: Accepted
- Date: 2026-07-19

## Context

Foundry Agent ServiceのFunction/Tool Callingは、Agentがツール呼び出しを要求した時点で
Runが `requires_action` 状態で一時停止し、呼び出し元が `submit_tool_outputs` するまで
再開しない。Agent Bの書き込み系操作（`CLAUDE.md` 2章・6章で「ask every time」＝承認必須と規定）
は、Slack上の承認ボタン押下がこの `submit_tool_outputs` のトリガーになる。

また、承認行為自体を「誰が実行できるか」（承認権限モデル）も合わせて決める必要があった。

### 実行方式の選択肢

- **A. 保留中のtool call情報（`run_id`, `thread_id`等）をTable Storageに保存し、ACAプロセスは
  いったん処理を終える。承認ボタンのSlackインタラクション受信時に保存済み`run_id`を使って
  `submit_tool_outputs`しRunを再開する**
- B. ACAプロセス内でRunをブロッキング待機させ続ける（asyncio.sleep loop等）

### 承認チャンネル・権限モデルの選択肢

- チャンネル構成: 全申請共通の1チャンネル固定 / 申請対象ごとに複数チャンネル
- 権限チェック: チャンネル参加を承認権限の証明とみなす / バックエンドに明示的な承認者リストを持つ

## Decision

- 実行方式は**選択肢A**を採用する。承認待ち状態はTable Storage（[[adr-005_storage-and-activity-log]]）
  に保存し、ACA再起動をまたいでも承認待ちを引き継げるようにする。却下時は空/エラーのtool outputを
  返してRunを終了させる
- 承認チャンネルは**1本の固定チャンネル**とする（申請対象ごとの分割はしない）
- 承認権限モデルは**「対象チャンネルへの参加＝承認権限」とし、バックエンド側で明示的な
  承認者ユーザーIDリストは管理しない**

## Consequences

- ACAの再起動・スケールイベントがあっても承認待ちのリクエストが消失しない
- チャンネル参加ベースの権限モデルは実装・運用がシンプルな反面、「誰かがそのチャンネルに
  追加されると自動的に承認権限を得る」という気づきにくい権限昇格経路になりうる。
  これは意図的に受容したリスクであり、`CLAUDE.md` 6章の最小権限方針からは一歩後退する判断である
- 将来、承認権限の厳密な管理が必要になった場合は、バックエンド側の明示的な承認者リスト
  （Table Storageの設定レコード等）への移行を検討する
