# ADR-008: Foundryスレッドのライフサイクル管理

- Status: Accepted
- Date: 2026-07-19

## Context

`CLAUDE.md` はユーザーID→`threadId`のマッピングを永続化する方針のみを述べており、
スレッドが無期限に伸び続ける前提だった。Agent A（相談チャット）はApp Home常設で
日常的に使われる想定のため、対話履歴が肥大化し続けると以下の問題が起こりうる。

- Foundry側のコンテキストウィンドウを圧迫し、古い話題が新しい回答の品質に悪影響を与える
- 無関係な過去の相談内容が新しい質問への回答に混ざる

## Decision

- ユーザーが明示的に新しいスレッドを開始できる **`/reset`相当のSlackコマンド**を用意する
- 加えて、一定期間（目安30日）操作が無いユーザーは、次回利用時に自動的に新規スレッドへ
  切り替える。最終利用日時は [[adr-005_storage-and-activity-log]] の `activity_log`
  （`chat_turn`レコード）から判定し、判定専用の追加データは持たない

## Consequences

- 手動リセットは実装コストが低い（Table Storageのマッピングレコード削除のみ）
- 自動リセットの判定を既存の`activity_log`から行うため、追加のデータ管理は発生しない
- 30日という閾値は暫定であり、実運用のフィードバックを見て調整する余地がある（未決事項）

## Review Note (2026-07-19)

PRレビューにて、Agent A（相談）とAgent B（申請）でthreadIdを共有するか分離するか曖昧との
指摘を受けた。文脈混在防止の観点では分離が望ましいが、本フェーズではシンプルさを優先し、
`CLAUDE.md`が元々想定する「ユーザーID→threadId」の1対1マッピング（Agentをまたいで共有）を
そのまま採用し、当該リスクは受容する。本ADRの決定を変更しない。文脈混在が実運用上の問題として
顕在化した場合は、本ADRをsupersedeする形でAgentごとのthreadId分離を別途起こすこと。
